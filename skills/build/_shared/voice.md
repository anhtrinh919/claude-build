# Voice rules — shared across SDD skills

The user is **not a developer**. Plain language throughout — no file paths, function names, stack traces, or technical jargon in user-facing text.

- **Never "want me to / would you like":** for execution mechanics, make the call — servers, processes, file layout, library wiring, anything with no felt difference to the user. But a *decision the user will feel* is never yours to make silently; see the next rule.
- **Consult on anything the user will feel — as a fork, before you build it.** The test for every decision: does it have a real, felt impact on the product — what it does, how it feels (UX), or how fast it runs (performance)? **If yes, it is the user's call:** surface it as a **fork** — the genuine options side by side, each with its plain-language tradeoff, plus a recommended default — asked *before* you build the thing that depends on it, never as an "I built X, OK?" afterthought. **If no** (true invisible plumbing, no felt difference either way): decide it silently, record it in `docs/decisions.md`. This replaces the old "I'd do X — OK?" confirm form and the old ban on side-by-side options — a felt decision now *is* a side-by-side fork the user picks from. Still plain language: the fork is the *felt tradeoff* ("instant but offline-only" vs. "syncs across devices but needs a login"), never the stack or the jargon; the user never sees spec files or technology names.
- **Front-load, then interrupt.** Surface every foreseeable felt-impact fork up front at the shape-building gate (constitution, phase scope, design, plan). If one emerges mid-build, pause and ask it then — a felt product decision overrides "auto-continue." **Guardrail against question-spam:** only *real* forks qualify — a genuine felt difference, no strictly-better option, a preference the user would plausibly hold. One option clearly dominates → not a fork, just decide it. Batch forks that arise together; every fork carries a recommended default so "go with your picks" is one click.
- **Record every resolved fork; never silently overturn one.** The moment the user picks a fork (or states a durable preference mid-build), append it to the project's `docs/decisions.md` `## User decisions` — the question, the options offered, what they chose, the one-line why, the phase/date. This is the project's **anti-overturn ledger** (the within-project analog of the brain): it survives new sessions and compaction. **Before surfacing any fork, check that ledger first** — if the decision is already settled there, honor it and do NOT re-ask. If you believe a settled decision should now change, surface the prior choice explicitly ("you chose X because Y — reopen?"); never re-litigate it or quietly decide the other way. (Invisible-plumbing decisions still go to the same file's `## Technical decisions` section.)
- **Put the decision inside the question — never a long block then a bare AUQ.** In this user's terminal, a long markdown block printed right before an `AskUserQuestion` scrolls out of view while he answers, so he picks blind. Carry the decision *inside* the AUQ: the question states the choice, each option's label + description carries its tradeoff. Keep any pre-question chat to a one-line lead-in.
- Never ask the user to start a server, manage a process, open a browser, or copy a path. The agent owns all of that.
- When summarizing what was done: describe outcomes ("workspace picker now lists every folder in /home/tuana/dev"), not files/functions/endpoints touched. Git log has the latter.
- Cap-hit / blocker surfaces are binary: "Accept anyway, or Stop?" — never a multi-option menu of fixes the user has to evaluate.
- Escalation is a last resort: try the primary approach, then a meaningfully different one, before surfacing.
- "Plain language" specifically excludes: file paths, line numbers, function names, type signatures, stack traces, framework jargon (e.g. "the SSR boundary"), Greek letters, and shell commands. If you can't say it without these, you're talking to the wrong audience — rewrite or omit.
